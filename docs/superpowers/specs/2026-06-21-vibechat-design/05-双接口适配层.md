# 05 · 双接口适配层（★ 技术核心）

> 技术性 40% 的命脉，也是严重扣分项②"只支持一种接口"的防御核心。本层让应用**同时讲 OpenAI 标准与 Anthropic 标准两种 API 格式**，并通过环境变量一键切换，上层业务零感知。

---

## 1. 赛题要求（复述）

- 必须支持 **OpenAI 标准接口**（`/chat/completions`）。
- 必须支持 **Anthropic 标准接口**（`/v1/messages`）。
- 两种接口通过**环境变量/配置切换**。
- README 必须说明两模式的配置（关键环境变量、模型名、启动方式）。

> 本项目的两路真实端点：OpenAI 标准 → **百炼兼容端点**（Qwen）；Anthropic 标准 → **DeepSeek `/anthropic` 兼容端点**（DeepSeek）。两路不同厂商、不同模型、各自讲对应标准格式。

---

## 2. 统一抽象

上层（情绪分析、AI 主持）只依赖一个接口，不关心背后是哪家、哪种格式：

```python
# backend/app/llm/base.py
from typing import Protocol, Optional

class LLMResponse:
    content: str                 # 文本内容
    raw_json: Optional[dict]     # 若要求结构化，解析后的 dict
    usage: dict                  # {input_tokens, output_tokens}
    provider: str                # "openai" | "anthropic"
    model: str

class LLMClient(Protocol):
    provider: str
    async def chat(
        self,
        system: str,
        user: str,
        *,
        json_schema: Optional[dict] = None,  # 要求结构化输出
        temperature: float = 0.3,
        max_tokens: int = 1024,
    ) -> LLMResponse: ...
```

业务调用示例：

```python
llm = get_llm()  # 工厂返回 openai 或 anthropic 实现
resp = await llm.chat(system=PROMPT, user=text, json_schema=EMOTION_SCHEMA)
emotion = resp.raw_json   # 两路都拿到同一个结构化 dict
```

## 3. 实现 A：OpenAI 标准（`OpenAIClient`）

### 3.1 请求

```python
# POST {OPENAI_BASE_URL}/chat/completions
{
  "model": settings.OPENAI_MODEL,                 # qwen-plus
  "messages": [
    {"role": "system", "content": system},
    {"role": "user",   "content": user}
  ],
  "temperature": 0.3,
  "max_tokens": 1024,
  "response_format": {"type": "json_object"}      # 要求 JSON（结构化时）
}
```

- Headers：`Authorization: Bearer {OPENAI_API_KEY}`。
- 百炼兼容端点：`OPENAI_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1`。

### 3.2 响应解析

```python
text = data["choices"][0]["message"]["content"]   # JSON 字符串
raw_json = json.loads(text)                        # 结构化时
usage = {"input_tokens": data["usage"]["prompt_tokens"],
         "output_tokens": data["usage"]["completion_tokens"]}
```

## 4. 实现 B：Anthropic 标准（`AnthropicClient`）

### 4.1 请求（关键：system 单独、content blocks）

```python
# POST {ANTHROPIC_BASE_URL}/v1/messages
{
  "model": settings.ANTHROPIC_MODEL,              # deepseek-chat
  "system": system,                               # ★ system 单独字段，不进 messages
  "messages": [
    {"role": "user", "content": user}
  ],
  "max_tokens": 1024,
  "temperature": 0.3
}
```

- Headers：`x-api-key: {ANTHROPIC_API_KEY}`、`anthropic-version: 2023-06-01`。
- DeepSeek 兼容端点：`ANTHROPIC_BASE_URL=https://api.deepseek.com/anthropic`。

### 4.2 结构化输出（Anthropic 无 `response_format`，用 prefill + 宽容解析）

Anthropic 标准不提供 `response_format`。采用**预填（prefill）**引导模型直接输出 JSON：

```python
# messages 末尾追加一条 assistant 预填，引导 JSON 续写
messages.append({"role": "assistant", "content": "{"})
# 在 system/user 中明确要求"只输出 JSON 对象"
# 解析时补回开头的 "{" 再 json.loads
```

- 兜底解析：若返回含多余文本，用正则提取首个 `{...}` 平衡块再 `json.loads`。
- ~~`tool_use` 增强~~（已砍：YAGNI）：只用 prefill 一种结构化方式，不引入 tool 强制调用，降低对 provider 能力的依赖。

### 4.3 响应解析

```python
# content 是 blocks 数组，取 type=="text" 的 text 拼接
text = "".join(b["text"] for b in data["content"] if b.get("type") == "text")
raw_json = parse_json_lenient(text)               # 补回 prefill 的 "{" + 宽容解析
usage = {"input_tokens": data["usage"]["input_tokens"],
         "output_tokens": data["usage"]["output_tokens"]}
```

## 5. 两路格式差异对照（让评审一眼看到"双标准真适配"）

| 维度 | OpenAI 标准 | Anthropic 标准 |
|---|---|---|
| 端点 | `/chat/completions` | `/v1/messages` |
| 鉴权头 | `Authorization: Bearer <key>` | `x-api-key: <key>` + `anthropic-version` |
| system 放哪 | `messages` 数组首条 `{role:"system"}` | **单独** `system` 字段 |
| 消息体 | `{role, content(str)}` | `{role, content:[{type:"text", text}]}` 或 str |
| 结构化输出 | `response_format={"type":"json_object"}` | **无** → prefill `{"` + 宽容解析 |
| 响应取值 | `choices[0].message.content` | `content[].text` 拼接 |
| usage 字段 | `prompt_tokens`/`completion_tokens` | `input_tokens`/`output_tokens` |
| 错误体 | `{error:{message}}` | `{type, error:{message}}` |

## 6. 配置与切换

### 6.1 环境变量（`.env`，README 必须写清这套）

```bash
# ===== 一键切换两种标准接口 =====
LLM_PROVIDER=openai              # openai | anthropic

# ----- OpenAI 标准（百炼 / Qwen）-----
OPENAI_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1
OPENAI_API_KEY=sk-***            # 占位，禁止入库
OPENAI_MODEL=qwen-plus

# ----- Anthropic 标准（DeepSeek /anthropic 兼容端点）-----
ANTHROPIC_BASE_URL=https://api.deepseek.com/anthropic
ANTHROPIC_API_KEY=sk-***         # 占位，禁止入库
ANTHROPIC_MODEL=deepseek-chat

# 其他
DATABASE_URL=sqlite:///./vibechat.db
LOG_LEVEL=INFO
```

### 6.2 工厂

```python
# backend/app/llm/factory.py
def get_llm() -> LLMClient:
    p = settings.LLM_PROVIDER
    if p == "anthropic":
        return AnthropicClient(settings.ANTHROPIC_BASE_URL,
                               settings.ANTHROPIC_API_KEY,
                               settings.ANTHROPIC_MODEL)
    return OpenAIClient(settings.OPENAI_BASE_URL,
                        settings.OPENAI_API_KEY,
                        settings.OPENAI_MODEL)
```

- 切换 = 改 `LLM_PROVIDER` 一个变量，**不改任何业务代码**。
- 依赖注入：FastAPI `Depends(get_llm)`，便于测试注入 mock。

## 7. 错误处理与降级

| 场景 | 处理 |
|---|---|
| 网络超时 | httpx timeout → 重试 1 次（指数退避）→ 仍失败 → `LLMError` |
| 4xx（鉴权/参数） | 不重试，`LLMError(code="auth"/"bad_request")`，前端明确提示"AI 服务配置异常" |
| 5xx / 限流 | 重试 1 次 → 仍失败 → `LLMError` |
| 结构化解析失败 | 重试 1 次 → 仍失败 → 情绪分析走**规则降级**（关键词推断向量，见 08），AI 主持走预设话术 |
| 全部失败 | 前端显示错误态 + "重试"按钮，**不白屏**（巡梦教训） |

> 降级是赛题硬性要求（"外部 LLM API 异常应有降级处理"），非临时补丁。

## 8. 可测试性（不调真 API 验证适配层）

- `OpenAIClient` / `AnthropicClient` 用 httpx + `respx`（或自写 fake transport）拦截请求：
  - 断言 **请求体构造正确**（OpenAI 的 `response_format`、Anthropic 的 `system` 单独 + prefill）。
  - 断言 **响应解析正确**（各自的 content/usage 取值 → 统一 `LLMResponse`）。
- `factory` 单测：`LLM_PROVIDER` 不同值返回不同实现。
- 业务层单测：注入 `FakeLLMClient`（固定返回预设 JSON），验证情绪分析/匹配/主持逻辑与真实 provider 解耦。
