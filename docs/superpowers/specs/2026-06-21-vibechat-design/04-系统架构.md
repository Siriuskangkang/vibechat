## 2. 技术栈选型与理由

| 层 | 选型 | 理由 |
|---|---|---|
| 前端框架 | **Next.js（App Router）+ React + TypeScript** | 赛题指定 Next.js；TS 提升工程质量分 |
| 样式 | **Tailwind CSS** | 暗色情绪美学自定义自由；零 UI 库依赖，演示稳（巡梦教训） |
| 可视化 | **纯 SVG/Canvas + CSS**（情绪色盘/雷达） | 避免重型图表库拖累；色盘逻辑简单，自绘可控 |
| 后端 | **Python FastAPI（async）** | 赛题指定；异步并发调 LLM + WebSocket 合适；Pydantic 类型契约 |
| ORM/DB | **SQLAlchemy 2.0 + SQLite** | 零外部依赖，演示不崩；可平滑切 PostgreSQL |
| 实时 | **WebSocket**（FastAPI 原生） | 房间群聊双向实时；广播即在线状态/消息/AI 主持推送 |
| 配置 | **pydantic-settings** 读 `.env` | 密钥不入库；类型化配置 |
| 进程 | uvicorn（开发）/ systemd 或 pm2（生产，见 08） | 与其他项目同范式 |

## 3. 分层架构与职责边界（直接对应"工程质量"40 分）

| 层 | 职责 | 禁止 |
|---|---|---|
| `routers/` | IO、入参校验、调用 service、统一响应 | 写业务逻辑、直接访问 DB |
| `services/` | 业务编排：情绪分析、匹配、AI 主持、降级 | 直接写 SQL、关心 HTTP |
| `ws/` | WebSocket 生命周期、房间连接池、广播 | 业务判断（交给 service） |
| `llm/` | 双接口适配、结构化输出、归一化（见 05） | 知道具体业务语义 |
| `repositories/` | 持久化（SQLAlchemy），封装查询 | 调 LLM、关心 HTTP |
| `core/` | config、异常基类、日志、降级开关 | 业务逻辑 |

**原则**：上层依赖抽象，不依赖具体实现；`LLMClient` 是接口，`OpenAIClient`/`AnthropicClient` 是实现（见 05）。

## 5. 实时通信（WebSocket）

### 5.1 连接与房间

- 客户端 `ws://host/api/ws/room/{slug}?session=...` 建连。
- `ConnectionManager` 维护 `{room_slug: set[WebSocket]}` 连接池；进房加入集合，离房移除。
- 进房广播 `{"type":"member_join","nickname":...}`；离房广播 `member_leave`。

### 5.2 消息协议（JSON）

```json
// 客户端 → 服务端
{"type":"message","content":"..."}
// 服务端 → 客户端（广播）
{"type":"message","sender":"失眠的云","avatar":{...},"content":"...","role":"user|ai|bot","ts":123}
{"type":"host","content":"...（AI 主持破冰/抛话题）...","ts":123}
{"type":"presence","online":3}
{"type":"room_mood","vector":[...],"resonance":0.82,"online":3}  // ★ 房间集体情绪场+共鸣度（见 10）
{"type":"summary","summary":"...","takeaway":"..."}   // 离场总结
{"type":"error","message":"..."}                       // 错误/降级提示
```

- `role` 区分 `user`(真人) / `ai`(主持) / `bot`(兜底)，前端样式区分（扣分项④反向证明 + mock 演示依据）。
- 所有消息同时 `INSERT messages` 落库，保证刷新/重连可拉历史。

### 5.3 历史与重连

- 进房 `GET /api/rooms/{slug}/messages?before=...` 拉最近 N 条历史。
- WS 断开自动重连（指数退避），重连后先拉历史补齐。

### 5.4 心跳与超时

- 客户端定期 ping；服务端超时清理僵尸连接，更新在线状态。

## 6. 配置管理（`core/config.py`）

- `pydantic-settings` 从 `.env` 读取，类型化（`Settings`）。
- 关键项：`LLM_PROVIDER`、双 provider 的 `BASE_URL/API_KEY/MODEL`、`DATABASE_URL`、降级开关、配额阈值。
- `.env` 被 `.gitignore`；`.env.example` 提交模板（仅变量名）。

---

## 7. 异常、日志、统一响应

- **统一响应结构**：`{code, message, data}` 或标准 HTTP + 错误体；错误码区分 `输入非法 / 模型失败 / 超时 / 降级`。
- **异常基类**：`VibeChatError` → 子类 `LLMError` / `MatchError` / `WSError`；全局异常处理器转统一响应。
- **日志**：关键步骤（情绪分析入参/出参、匹配结果、WS 事件、provider 调用耗时）结构化日志；演示期 INFO 即可。
- **可观测**：LLM 调用耗时埋点（服务"时间效率"分）。

## 8. 工程质量要点（拿"技术性 40%"）

- **类型契约**：Pydantic schemas（请求/响应/LLM 输出）贯穿前后端；前端 `types.ts` 与之一致。
- **纯函数单测**：向量构造、加权 cosine、同频度、双接口响应归一化、匹配排序/平局（见 [09](./09-测试验收与演示.md)）。
- **依赖注入**：`get_llm()` / `get_session_repo()` 注入，便于 mock 测试。
- **provider 可替换**：上层只依赖 `LLMClient` 接口，切换/测试不波及业务（见 05）。
- **最小依赖**：能用标准库/单依赖解决就不引三个（巡梦教训）。
