## 3. 评审维度 → 设计抓手

| 维度 | 权重 | 判分要点 | 本项目抓手 |
|---|---|---|---|
| **功能性** | 40% | 情绪识别、匹配、匿名聊天、线上演示全部实现，核心流程稳定 | 7 项必做全覆盖；情绪→匹配闭环可量化展示；降级保证稳定 |
| **技术性** | 40% | 双标准适配清晰、前后端架构合理、工程质量/可维护性/异常处理/部署完成度高 | 双接口统一抽象层；分层架构（router/service/ws/llm/repo）；类型契约；统一异常；单测核心纯函数 |
| **用户体验** | 20% | UI 美观、交互顺畅、情绪表达准确、匿名氛围自然、有继续使用吸引力 | 情绪色盘可视化、房间色场、匿名身份生成、AI 主持破冰降低冷启动门槛 |

## 4. 严重扣分项 → 对策（重点防御）

| 扣分项 | 为什么会被扣 | 本项目对策 |
|---|---|---|
| **① 无线上演示地址 / 地址无法访问** | 没部署或挂了 | 提前部署到 kang 服务器，赛前排烟测；README 首屏放演示链接 + 测试账号；nginx + SSL 复用现有证书，WS 配 Upgrade 头 |
| **② 只支持 OpenAI 或只支持 Anthropic** | 双标准没做全 | 适配层**双实现都真实跑通**（OpenAI→百炼，Anthropic→DeepSeek `/anthropic`），均经单测；README 明确两模式配置与切换命令；演示中**现场切换两种 provider 各跑一次** |
| **③ 情绪分析与匹配脱节（只贴标签未影响匹配）** | 情绪只是装饰 | 情绪向量**实打实参与匹配**：cosine 相似度决定进哪间房，并把"同频度 X%"**显式展示**给用户；`emotion_analyses` 表留痕可追溯；色盘可视化把"为什么进这间房"画出来 |
| **④ 匿名聊天无法收发** | 实时通信没通 | WebSocket 房间广播，发送即广播、即落库、即展示；mock 多用户脚本验证多端收发；断线自动重连 |

## 6. 与第二期巡梦的方法论复用

本项目直接复用巡梦经验沉淀（`docs/espoirdream-main/经验沉淀.md`）中可迁移的条目：

- 定一个"感觉"，技术栈为它服务（暗色情绪社交）。
- 少依赖、高完成度 > 多技术栈（SQLite + 最小依赖）。
- 卖情绪价值，不卖功能清单。
- 动手前判断 workflow vs agent：情绪匹配与 AI 主持路径清晰 → **用确定性 workflow，不上 agent**。
- 提示词写成可被代码消费的 JSON schema 契约。
- 每个 AI 环节挂降级，无 key 也能演示。
- provider 配置化、安全三件套（哈希 / 隔离 / 配额）。
- 交付：在线 demo + 默认账号 + 免登录 + 文档齐套。
- 避坑：重构删干净死代码、补核心纯函数单测、提示词补 few-shot、降级模式 UI 明确标注。
