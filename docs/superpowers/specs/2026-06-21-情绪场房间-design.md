# 情绪场房间（会呼吸的房间页）· 设计

> 日期：2026-06-21 ｜ 状态：待评审
> 定位：兑现 [`10-情绪场差异化设计`](./2026-06-21-vibechat-design/10-情绪场差异化设计.md) 的「会呼吸的情绪房间」立意——当前房间页是脱离主题的通用聊天壳，本设计把它改造成会呼吸、可感知同频的情绪场。
> 赛题映射：开放发挥「情绪可视化 / 产品气质」；评分体验维度 20%（「匿名社交氛围自然」「有继续使用的吸引力」）。
> 依据：`app/room/[slug]/page.tsx`、`components/*`、`app/globals.css`、`ws/rooms_ws.py`、`ws/connection_manager.py`、`services/mood_field.py`。

---

## 1. 背景与问题（单调的根因）

房间页 `room/[slug]/page.tsx` 是标准三段式（header / 消息流 / 输入框）。问题不在丑，而在：

- **已有情绪可视化资产散落各处、房间页几乎没用**：`EmotionColorWheel`（涟漪+光晕落点）只在 `/analyze`；`MoodJourney`（弧线）只在离场；`StarField`（星尘）只在首页，**房间页未挂载**。
- **后端实时数据被浪费**：`room_mood` / `resonance` 每次进出房都在算并推送，前端只把它显示成 `共鸣度 65%` 一个干巴巴的数字（`ResonanceBadge`）。
- 结论：房间页 = 脱离「情绪社交」主题的通用聊天壳，与 spec 10 的「会呼吸的情绪房间」立意脱节。

本设计目标：**把房间页改造成会呼吸、可感知同频的情绪场**，让「情绪驱动社交」在对话过程中持续可见。

---

## 2. 设计原则：克制爆发

用户诉求「越显眼越好」，但**显眼 ≠ 全程高饱和**。三条原则：

1. **克制基调 + 瞬间爆发**：全程维持现有「深夜情绪星云」克制美学（暖白 `#ECE9E2` + 暖金 `#C9A86A` + 低饱和 + `var(--ease)`）；把「显眼」集中到关键瞬间的高对比爆发（共振时刻 / 发言涟漪 / 共鸣光点）。关键时刻的安静爆发，强过全程热闹，也贴赛题「氛围自然」。
2. **单一焦点**：同一时刻只允许一种爆发（共振 / 涟漪 / 光点互斥排队），避免视觉过载。
3. **零新设计语言**：所有新组件**只复用现有 `@theme` token 与 keyframes**（`breathe/glow/ripple/pulseScale/fadeUp`），不引组件库、不引图表库（严守 [README §3.5](./2026-06-21-vibechat-design/README.md) 轻量化红线）。

---

## 3. 与现有设计的关系（衔接 spec 10，不违反轻量化红线）

| spec 10 已定义 | 本设计在房间页的兑现 |
|---|---|
| ① 房间集体情绪场 `room_mood` → 背景色 | 深化为 `EmotionWeather`（天气意象）+ `RoomNebula`（成员星云） |
| ② 共鸣度 `resonance` | `ResonanceBadge` 保留，新增 `ResonancePulse`（破阈高光） |
| 共享底座：成员向量聚合（不每条消息调 LLM） | **完全沿用**——本设计所有新增都不调 LLM、不新增 DB 表 |

**轻量化核对**（对齐 spec 10 §7）：`RoomNebula`=纯 Canvas、`EmotionWeather`/`MoodStream`=纯 SVG/CSS、涟漪/光点/脉搏=纯 CSS 动画、`typing`/`reaction`=纯 WS 事件。**零新增依赖、零额外 LLM 调用、零新表。**

## 4. 数据与协议地基（WebSocket 扩展）

现状 WS 事件：`presence`（仅 `online` 数）/ `room_mood` / `member_join` / `message` / `host` / `history`。交互层与星云依赖以下扩展（均为 `rooms_ws.py` + `connection_manager.py` 小改，沿用现有 broadcast 路径，向后兼容——前端未识别事件静默忽略）：

| 方向 | 事件 | 载荷 | 用途 |
|---|---|---|---|
| S→C | `presence`（升级） | `{type, online, members:[{id,color,shape}]}` | 星云着色（manager 已存每人 vector，附 `emotion_color` 算出的 color + shape） |
| C→S | `typing` | `{type:"typing"}` | 输入时上报 |
| S→C | `typing` | `{type:"typing", color, nickname}`（exclude 发送者） | 情绪脉搏输入提示 |
| C→S | `reaction` | `{type:"reaction", target_ts, color}` | 送共鸣光点 |
| S→C | `reaction` | `{type:"reaction", color, target_ts, nickname}`（exclude 发送者） | 光点飘向目标气泡 |

> `connection_manager` 现状 `members(slug) -> {ws: {nickname, avatar, vector}}` 已含所需信息（color/shape 在 avatar 里）；只需在 `_broadcast_state` 的 presence 帧里把成员快照序列化出去。
> **`id` 来源与匿名**：presence 里的 `id` 用 `session_id` 的短哈希（md5 前 6 位），**不直接广播 session_id**，仅供前端 `nebulaPos(id)` 稳定定位星点；`color`/`shape` 复用 avatar，不新增信息暴露。

## 5. 三层设计

### 5.1 背景层（看得见人 + 共振高光）

**`RoomNebula`**（升级 `StarField` 为「成员星云」，纯 Canvas）：
- 每个在线成员 = 一颗带其情绪色（`avatar.color`）的漂浮光点；位置由成员 id 哈希稳定 + 缓慢漂移（沿用 `StarField` 的 `floatSlow` 节奏）。
- 粒子数沿用 `N = min(70, w*h/22000)` 上限；成员多时按情绪聚类排布。
- 触发响应：本人/他人发言时对应星点 `pulseScale` 一次；成员进出淡入淡出。
- 与全局 `body::before` 星云背景叠加，`pointer-events:none`，`-z-[1]`。

**`ResonancePulse`**（共振时刻，纯 CSS）：
- 监听 `resonance`：连续 ≥ `0.7` 且持续一个采样窗口 → 触发一次全屏柔光脉冲（`radial-gradient` + `animate-glow`）+ 一行宋体「此刻，你们同频」（`--font-display`）。
- 冷却 4s，避免廉价；与发言涟漪/光点遵循单一焦点排队。

### 5.2 信息层（把干巴巴的数字变成画面）

**`EmotionWeather`**（header，与 `ResonanceBadge` 并列，纯 SVG/CSS）：
- 把 `room_mood` 的 `[valence, arousal, ...]` 映射成动态天气意象，由纯函数 `weatherOf(valence, arousal)` 决定（见 §9 单测）：

  | valence | arousal | 意象 | 色调 |
  |---|---|---|---|
  | ≥+0.2 | ≥0.55 | 放晴 | 暖金高光 |
  | ≥+0.2 | <0.55 | 晨光 | 柔暖低饱和 |
  | ∈(-0.2,+0.2) | * | 薄云 | 中性青灰 |
  | <-0.2 | <0.45 | 微雨 | 冷蓝低沉 |
  | <-0.2 | ≥0.45 | 骤雨 | 冷紫动荡 |

  边界规则（消除歧义，见 §9 单测）：`valence ≥ +0.2` → 放晴/晨光；`valence ≤ -0.2` → 微雨/骤雨；其间 `(-0.2, +0.2)` → 薄云（不看 arousal）；正/负向内再按 arousal `0.55`/`0.45` 分高低能。

- 极简符号 + 一句诗意短文案（如「房间微雨 · 低沉」），比 `共鸣度 65%` 直观。

**`MessageBubble` 增强**：真人气泡加一层极淡情绪色 `box-shadow` 光晕（发送者 `avatar.color`，`rgba α≈0.18`）；消息流颜色随对话情绪流动。

**`MoodStream`**（消息区左侧 2px 细条，纯 CSS）：
- 随消息累积的情绪色带（取每条消息发送者色），像情绪心电图，可视化对话整体走向。低优先，空间不够可省。

### 5.3 交互层（温柔差异化）

**发言涟漪**：`send` 时从输入区扩散一圈 `animate-ripple` 到背景——「你的情绪扰动了这个场」。

**`TypingPulse`**（替换冷冰冰的「…」）：收到他人 `typing` 时，footer 上方显示该人情绪色 `animate-pulse-scale` 点 + 极小昵称；停止输入 2s 后淡出。

**共鸣光点**（`ReactionLayer`）：
- 触发：移动端长按 / 桌面端悬停 0.4s → 弹出单颗「接住你」光点 → 发 `reaction`。
- 收到 `reaction` 时，一颗该色光点沿贝塞尔弧线飘向 `target_ts` 对应气泡，聚成 1.5s 微光后淡出。
- **刻意不做成点赞计数**（不显示数字、不可堆叠攀比），避免破坏匿名氛围——这是产品气质取舍。

## 7. 视觉规范（复用现有 `@theme` / keyframes）

- 色彩：仅用 `--color-ink/ink-dim/ink-faint/accent/accent-soft/surface/line`；情绪色统一走 `lib/color.ts:emotionHsl`。
- 动效：仅用 `animate-breathe/glow/ripple/pulse-scale/fade-up/fade-in/msg-in` + `var(--ease)`；新增动画必须先确认无法复用再考虑加 keyframes。
- 字体：标题/诗意文案用 `--font-display`（宋体）；正文 `--font-sans`。
- `prefers-reduced-motion`：全部降级为静态（`globals.css` 已有全局 media query 覆盖，自动生效）。

## 8. 错误处理与降级

- **Canvas 性能**：`RoomNebula` 限粒子数；`requestAnimationFrame` 在页面不可见时暂停；低端设备由 reduced-motion 兜底。
- **协议向后兼容**：前端未识别的新事件静默忽略；后端缺失新事件时，星云用 init 时本地已知 vector、`TypingPulse`/光点不出现——**核心聊天流程零回归**。
- **聚合异常**：`room_mood`/`resonance` 异常时，`EmotionWeather` 退回中性「薄云」、`ResonancePulse` 不触发（沿用 spec 10 §8 兜底：退回固定主题色场）。
- **光点目标丢失**：`target_ts` 找不到对应气泡时，光点在屏幕中下方淡出，不报错。

---

## 9. 测试策略

- **纯函数单测（pytest 同构 / vitest）**：
  - `weatherOf(v,a)`：覆盖五象限边界（含 ±0.2、±0.45/0.55 阈值）。
  - `nebulaPos(id)`：同 id 稳定、不同 id 分散。
  - 共振触发判定：连续破阈才触发、冷却期内不重复。
- **WS 契约**：`typing`/`reaction` 广播 exclude 发送者、presence 含 members 快照（沿用 `test_connection_manager.py` 的 FakeWS 模式）。
- **回归**：进房/收发/离场流程不变；reduced-motion 下全部静态。
- **演示手测**：开 2 个无痕窗口模拟多人，验证星云多色点、共振触发、光点跨客户端飘达。

## 10. 验收标准

- [ ] 进房即见成员情绪星云 + 房间情绪天气

- [ ] 他人输入时显示情绪色脉搏点
- [ ] 真人气泡带发送者情绪色光晕
- [ ] `prefers-reduced-motion` 下全部降级为静态
- [ ] 现有进房/收发/离场流程零回归；`pytest tests/ -v` 全绿

## 11. 风险与取舍

| 风险 | 对策 |
|---|---|
| 三层同屏视觉过载 | 克制爆发 + 单一焦点排队（§2.2） |
| 星云 Canvas 吃性能/电量 | 粒子上限 + 不可见暂停 + reduced-motion 兜底 |
| 共鸣光点变味成点赞 | 刻意无计数、不可堆叠（§5.3） |
| 天气意象太主观 | 映射走纯函数 + 五象限单测，可调但确定 |
| WS 协议扩展破坏现有 | 新事件向后兼容、前端静默忽略（§8） |

## 12. 与 M9 / 登录决策的关系

- 本设计是纯前端 + 轻量 WS 扩展，**与 [M9 优化冲刺](../plans/2026-06-21-vibechat/M9-优化冲刺-活跃房间与安全闭环.md) 正交**（M9 偏后端逻辑：安全闭环/活跃房间/匹配升级），两者可并行或前后落地。
- 仍沿用「不做账号登录」结论（cookie session + 系统生成匿名身份），`typing`/`reaction` 事件基于现有 `session_id`/`nickname`，不引入账号。
